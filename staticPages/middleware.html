Middleware Ordering in ASP.NET Core

Key Concept:

  • Request flows Top → Bottom
  • Response flows Bottom → Top

Real-Life Analogy

Entering a company:

  • Security Check (Authentication)
  • Reception (Routing)
  • Manager (Controller)
  • Exit Process (Response)

Correct Middleware Order

var app = builder.Build(); app.UseExceptionHandler("/error"); app.UseHttpsRedirection(); app.UseStaticFiles(); app.UseRouting(); app.UseAuthentication(); app.UseAuthorization(); app.MapControllers(); app.Run();

Request Flow

  • Exception Handler
  • HTTPS Redirection
  • Static Files
  • Routing
  • Authentication
  • Authorization
  • Controller Execution

Wrong Order Example

app.UseAuthorization(); app.UseAuthentication();

❌ Problem: Authorization runs before Authentication

Correct Order

app.UseAuthentication(); app.UseAuthorization();

✔️ Authentication runs before Authorization

Another Common Mistake

app.UseRouting(); app.UseEndpoints(...); app.UseAuthentication(); // WRONG

❌ Authentication should be before endpoint execution

Interview Answer

Middleware executes in the order registered for requests and in reverse order for responses. Correct ordering is critical—for example, authentication must come before authorization, and routing must come before endpoint execution.

Pro Tips

  • Static files can short-circuit the pipeline
  • Exception handler should be at the top
  • Custom middleware placement is important
